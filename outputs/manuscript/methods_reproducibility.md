# Methods (Reproducibility appendix to Sections 2.1-2.5 of the main Perspective)

This document is the version-pinned, parameter-pinned, container-pinned
reproducibility appendix to the Methods of *A sequence- and structure-only
computational pipeline that approximates the post-trial composite correlate
proposed by Estofolete et al. for dengue vaccines* (Weatherhead et al.,
intended for *npj Vaccines*). All numbers reported in the main text were
generated by the artifact at git tag `v1.0-dengue-results` of
https://github.com/JamesWeatherhead/immunoinformatics_platform-dengue.

---

## 2.1 Pipeline overview

The pipeline is a fork of the Versiani/McCaffrey
`immunoinformatics_platform` (Versiani et al. 2026 *Sci Adv*
12:eaeb2066) implemented as a Nextflow DSL2 workflow, with every
process executed inside a pinned Apptainer/Singularity image.
Inputs (DENV-1 to DENV-4 reference polyproteins from UniProt and the
parent-strain FASTAs of the three Phase 3 vaccine constructs) flow
through CD-HIT redundancy clustering, two parallel B-cell tracks
(EpiDope sequential prediction and BepiPred-3.1 sequential
prediction, plus an AlphaFold-Multimer + DiscoTope-1.1 geometric
track for the envelope homo-dimer), two parallel T-cell tracks
(IEDB MHC Class I prediction tools 3.1.2 and IEDB MHC Class II
prediction tools 3.1.6, both invoking NetMHCpan-4.1 / NetMHCIIpan-4.1),
and an IEDB Population Coverage stage that consumes AFND release
2024-12 HLA frequencies. Per-residue B-cell scores and per-peptide
T-cell binding ranks are post-processed by `scripts/dengue/quick_score_tiers.py`
into the four Estofolete Tier A / Tier B sub-scores, restricted to
the EDE residue window (Section 2.5). The full chain is orchestrated
unattended on the cluster by `scripts/dengue/cluster_master_pipeline.sh`,
which is idempotent across phases M1 to M12.

## 2.4 Parameters

Defaults below refer to the upstream `immunoinformatics_platform` defaults
(Versiani et al. 2026) where applicable. "This fork" means the value is
set by `scripts/dengue/cluster_master_pipeline.sh` or
`scripts/dengue/quick_score_tiers.py` and may differ from the upstream
default.

| Process | Parameter | Value | Default? | Justification |
|---|---|---|---|---|
| Apptainer (all processes) | `runOptions` | `--no-home` | NOT default | Upstream automounts host `$HOME`, which shadows older containers' pip site-packages with host numpy>=1.24 and triggers a `np.bool` AttributeError in pandas 1.4 era containers (e.g. `cdhit_to_tsv`). `--no-home` keeps Nextflow's workdir mount but blocks the home-mount. |
| Apptainer | `autoMounts` | true | default | Allows Nextflow to bind the work directory. |
| CDHIT (input proteins) | `--cdhit_input_proteins` | yes | default | Cluster input polyproteins for redundancy. |

| BepiPred-3.1 (B-cell) | (run inside `iedb-b-cell` container, default thresholds) | upstream defaults | default | We do not consume BepiPred per-residue thresholds in `quick_score_tiers`; included for cross-validation only. |
| DiscoTope-1.1 | discotope score cutoff | -7.7 | this fork (`quick_score_tiers.py`) | DiscoTope 1.1 reports a propensity-corrected log-odds; the canonical "epitope" cutoff in the original Andersen et al. 2006 paper is -7.7 (sensitivity ~0.40, specificity ~0.95). |

## 2.5 EDE residue list (verbatim)

The Envelope Dimer Epitope (EDE) is the quaternary epitope on the
DENV E homo-dimer first characterised by Rouvinski et al. 2015
(*Nature* 520:109-13, PMID 25849258). The residues participating in
the EDE-1 / EDE-2 paratope contact (E protein numbering, 1-indexed
within mature E starting at the polyprotein E-protein boundary which
in our coordinates is polyprotein position 281) are encoded
verbatim in `scripts/dengue/quick_score_tiers.py` lines 201-206 as:

```
EDE_E_RESIDUES = (
    range(67, 77)                        # 67 68 69 70 71 72 73 74 75 76
  + range(99, 125)                       # 99 100 101 ... 124
  + [79, 80, 84, 86, 88, 91, 152, 153]
  + [246, 248, 249, 251]
)
```

In polyprotein coordinates (E_start = 281, so polyprotein_pos =
E_pos + 280) these translate to positions 347 to 356, 379 to 404,
plus 359, 360, 364, 366, 368, 371, 432, 433, 526, 528, 529, 531.
Restriction to this set is the practical operationalisation of
"EDE-restricted" Tier A.1 and Tier A.2 sub-scores in the main text.

GAP: the residue list above is the implementation we shipped at
`v1.0-dengue-results`. It is consistent with the residues shown in
Rouvinski 2015 Figures 1 and 4 (b-strand `b` to `c` of domain II,
the fusion-loop adjacent residues, and the cd-loop and lateral-ridge
residues at the inter-protomer contact) but is not a verbatim copy
of any single supplementary table. The exact line-of-table-from-Rouvinski
correspondence will be added as `docs/dengue/ede_residue_provenance.md`
before journal submission.

## 2.6 Statistical analysis

### 2.6.1 Composite scoring formula

For each input proteome `s`, the four Estofolete sub-scores
(Section 2.3 of the main text) are normalised to [0, 1] by
serotype-cohort min-max with a fixed lower floor:

  - `tier_a1(s) = min(1, epidope_count_EDE(s) / max(50, max_serotype_count))`
  - `tier_a2(s) = min(1, discotope_count(s) / max(50, max_serotype_count))`
  - `tier_b2(s) = min(1, netmhcpan_i_strong_binders_EDE(s) / max(100, max_serotype_count))`
  - `tier_a(s) = mean(tier_a1, tier_a2)` if `tier_a2 > 0`, else `tier_a1`
  - `tier_b(s) = tier_b2`
  - `composite(s) = w_A * tier_a(s) + w_B * tier_b(s)`

with `w_A + w_B = 1`. Tier B.1 (cross-serotype MBC breadth) is
reported as `post_phase1_required` (string) and excluded from the
composite, consistent with Section 2.5 of the main text.

### 2.6.2 Weight selection

Two weight pairs are reported:

  - **Pre-registered equal-weight (primary).** `w_A = 0.5, w_B = 0.5`.
    This is the weight pair we registered before unblinding the Phase
    3 efficacy comparison. It carries no test-set leakage.
  - **Exhaustive-sweep (secondary).** `w_A = 0.25, w_B = 0.75`. We
    swept `w_A` over `{0.0, 0.05, 0.10, ..., 1.0}` against the n=3
    Phase 3 efficacy test set and selected the pair maximising
    Pearson r. Because this is a 21-value sweep on a 3-point test
    set, the optimised r is upward-biased; it is reported only as a
    secondary, transparency sensitivity analysis.

The headline Pearson r = 0.45 (n = 3) reported in Section 3.3 of
the main text uses the pre-registered 0.5/0.5 weights. The 0.25/0.75
sweep result is reported alongside as a sensitivity analysis with an
explicit note that it is in-sample.

### 2.6.3 Bootstrap confidence intervals

For any reported correlation `r` between composite score and
published Phase 3 efficacy:

  - Resample the n = 3 program-level points with replacement.
  - Compute Pearson r on each resample.
  - Repeat for `N = 10,000` resamples.
  - Report the 2.5th and 97.5th percentiles (percentile method;
    Efron 1979 *Ann. Statist.* 7:1-26).

Because n = 3, the bootstrap CI is wide and we explicitly mark it
`[CI: n<4]` in the manuscript table. The bootstrap is reported for
methodological completeness, not as a basis for inference.

### 2.6.4 Permutation test

To assess whether the observed correlation could plausibly arise by
chance under the null of no association:

  - Hold the predicted composite scores fixed.
  - Shuffle the published Phase 3 efficacy labels across the three
    programs (`N = 10,000` shuffles of {56.5%, 61.2%, 79.6%} across
    {CYD-TDV, TAK-003, Butantan-DV}).
  - Compute Pearson r for each shuffle.
  - Empirical p = (#shuffles with |r| >= |r_observed|) / 10,000.

With only 3! = 6 distinct label permutations, the minimum achievable
p-value is 1/6 ≈ 0.167; any p reported is interpretable only as a
descriptive bound, not a hypothesis-test rejection.

### 2.6.5 Rank correlation and leave-one-out

In addition to Pearson r we report:

  - **Spearman ρ** on the same three program-level points.
  - **Leave-one-vaccine-out** cross-validation: drop each vaccine in
    turn, refit no parameters (composite is unweighted at 0.5/0.5),
    and report the rank order of the held-out vaccine relative to the
    other two.

## 2.7 Reproducibility

### 2.7.1 Hardware

  - Compute host: `hsv-server` (10.176.2.135), University of Texas
    Medical Branch.
  - Accelerators: 8 x NVIDIA A100 80 GB SXM4 (CUDA 12.x driver,
    NVLink-coupled).
  - Storage: ~5 TB locally attached for AlphaFold databases; outputs
    on `/data/james/ica-dengue/`.
  - All AlphaFold-Multimer jobs run serially with all 8 GPUs assigned
    per job (`CUDA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7`); see Section 2.4.

### 2.7.2 Random seeds

A global random seed of **42** is set throughout:

  - `numpy.random.seed(42)` at the top of every Python helper script.
  - AlphaFold-Multimer uses its default per-model seeding; we do not
    override the AlphaFold seeds and instead rely on the
    `--use_precomputed_msas=true` flag plus pinned databases to
    deliver bit-identical structures across re-runs.
  - The bootstrap and permutation procedures (Section 2.6) are
    seeded with 42 in the harness script
    `scripts/dengue/retrospective_phase3_validation.py`.

### 2.7.3 Container builds

All Apptainer/Singularity SIFs were built directly from Docker
images on a local Docker daemon, using:

```
apptainer build <name>.sif docker-daemon://<image>:<tag>
```

with Apptainer 1.x. The Dockerfiles (committed at the release tag)
and the resulting SIFs (on `hsv-server`) are the canonical artifacts.

### 2.7.4 Code

  - Repository: https://github.com/JamesWeatherhead/immunoinformatics_platform-dengue
  - Release tag: `v1.0-dengue-results`
  - Pedigree: fork of Peter McCaffrey's
    `pmccaffrey6/immunoinformatics_platform` (Versiani et al. 2026
    *Sci Adv* 12:eaeb2066).
  - Orchestration entry-point: `scripts/dengue/cluster_master_pipeline.sh`.

### 2.7.5 Re-running the pipeline

```
git clone https://github.com/JamesWeatherhead/immunoinformatics_platform-dengue
cd immunoinformatics_platform-dengue
git checkout v1.0-dengue-results
# Build SIFs (docker images must be present on local daemon)
bash scripts/dengue/cluster_build_netmhcpan.sh
# Run the autonomous chain
tmux new -s master_pipeline -d \
  "bash scripts/dengue/cluster_master_pipeline.sh; sleep 604800"
```

Total wall time on hsv-server: approximately 36 to 48 hours
(dominated by 4 serial AlphaFold-Multimer runs at ~4 h each).
